## 1. 💡 도입부: 철학은 하나인데, 메뉴는 둘이었다

PriSincera의 성장 철학은 처음부터 하나의 문장이었습니다 — **배움이 실행이 되고, 실행이 복기를 거쳐 다음 배움을 개인화한다.** 매일의 지식 큐레이션(Daily Digest)과 주간 실행·회고 트래커(Pace Note)는 이 플라이휠의 앞바퀴와 뒷바퀴로 설계된 서비스였습니다.

문제는, 사용자에게 이 둘이 **완전히 분리된 두 개의 메뉴**로 보였다는 것입니다. 다이제스트에서 좋은 아티클을 읽어도 그것을 실행으로 옮기는 동선이 없었고, 궤도(Orbit)를 완주해도 그 경험이 다음 날의 배움에 반영된다는 사실을 화면 어디에서도 체감할 수 없었습니다.

역설적이게도 **백엔드는 이미 연결되어 있었습니다.** 성장 루프 파이프라인(Growth Loop Phase 0~4)이 완료·회고 신호를 추천 엔진과 다이제스트 렌즈로 환류하고 있었으니까요. 남은 것은 단 하나 — **이 보이지 않는 연결을 화면 위로 끌어올리는 마지막 표면 작업**이었습니다.

> 💡 **한 줄 요지**: 통합의 본질은 신규 기능 개발이 아니라, 이미 존재하던 데이터 플라이휠에 "하나의 여정"이라는 UI를 입히는 일이었습니다.

## 2. 🏷️ 이름 짓기: Re-Learn, 그리고 Re-run

통합 서비스의 이름은 **ReLearn(리런)** 으로 확정했습니다. "**다시 배우다(Re-Learn)**"와 "**다시 달리다(Re-run·러너)**"의 이중 의미를 한 단어에 담아, 서비스의 타깃인 *"제로부터 다시 배우는 러너(Learner & Runner)"* 를 이름 자체가 상징하도록 했습니다.

> **태그라인**: "매일 제로에서, 다시 배우고 다시 달린다 — Learn from zero, run again."

ZeroRun·Day One·Base Camp 같은 후보도 검토했지만, "통합"이라는 사건 자체를 이름이 설명하는 것은 ReLearn뿐이었습니다.

## 3. 🧭 전략 결정: 병행이 아니라 승계(Successor)

가장 중요한 의사결정은 기술이 아니라 **포지셔닝**이었습니다.

처음에는 기존 두 서비스를 그대로 두고 리런을 병렬 추가하는 "추가형(Additive)" 전략으로 출발했습니다. 그러나 계획을 다듬으며 이를 재정의했습니다 — **리런은 두 서비스의 승계자(Successor)이며, "추가형"은 영구 병행이 아니라 전환기의 무파괴 이행 전략이다.**

이 재정의가 만들어낸 세 가지 원칙:

1. **패리티 게이트 없이는 일몰 없음** — 구독 해지 자체화, 아카이브 열람, 캘린더·옴니 검색의 폐기/이관 판정까지 기능 패리티 체크리스트(P1~P6)를 전부 해소한 뒤에만 기존 메뉴를 내린다.
2. **노출 종료 ≠ URL 삭제** — `/daily/:date` 같은 콘텐츠 상세 URL은 수개월간 쌓인 SEO 자산입니다. 메뉴에서 내리더라도 URL은 영구 보존하고, 구 경로는 서버 301로 새 주소에 승계합니다.
3. **검증 후 전환은 마지막에** — 신규 라우트 `/relearn`을 먼저 열고, 안정화를 확인한 뒤에야 진입점을 교체합니다.

## 4. 🛠️ 아키텍처: 신규 API 0, 신규 잡 0

통합 규모의 프로젝트에서 가장 자랑하고 싶은 숫자는 화려한 신규 기능이 아니라 **0**입니다.

### 1) 방향 분리 — 파이프라인 무접촉

배치 파이프라인(Cloud Run Jobs)은 콘텐츠를 **생성(쓰기)** 하는 경로이고, 리런은 기존 API를 **읽고** 기존 사용자 쓰기 API를 재사용할 뿐입니다. 신규 API 0건, 신규 잡 0건, `cloudbuild.yaml` 변경 0건. 리런의 빌드가 실패해도 새벽 파이프라인은 기존 이미지로 정상 가동됩니다.

### 2) 추천 통계 오염 방지

리런의 핵심 UX인 "궤도로" 원클릭은 커스텀 궤도(`custom-`)를 대량 생성할 수 있습니다. 이것이 추천 풀의 velocity·퇴출 통계를 왜곡하지 않도록, 파이프라인 통계 집계에서 커스텀 궤도를 명시적으로 제외했습니다.

```javascript
// pacenote-composer — 리런發 커스텀 궤도는 추천 풀 통계에서 제외
const poolTasks = weekTasks.filter(task => !task.id.startsWith('custom-'));
```

### 3) 모놀리식 해체 — 추출이 곧 부채 상환

배움 카드는 구 DailyDigest 페이지에, 실행·회고는 약 1,900줄의 PaceNoteDashboard에 한 덩어리로 묶여 있었습니다. 리런 셸에 조립하려면 먼저 **섹션 컴포넌트로 추출(Phase B-0)** 해야 했는데, 여기서 원칙을 하나 세웠습니다 — **추출한 컴포넌트는 기존 페이지도 함께 소비하도록 치환한다.** 복사가 아니라 단일화이므로, 통합 작업 자체가 코드 부채를 줄이는 방향으로 작동했습니다.

### 4) 중복 페치 금지

셸이 배움·실행 스테이지를 함께 그리므로, 프로필·궤도·트랙 피드는 **셸 레벨에서 1회만 페치**해 하위로 전달합니다. 스테이지마다 각자 호출하는 순간 같은 API가 화면당 3번 불리는 구조가 되기 때문입니다.

## 5. 🔄 하나의 여정: 오늘 | 기록, 그리고 3-Stage 루프

리런의 화면은 단일 라우트 안에서 **`오늘 | 기록`** 두 뷰를 전환합니다.

```
┌─ ReLearn (오늘) ─────────────────────────────────────┐
│  ─ 성장 루프 리포트 (완료율 · 연속 · 집중 도메인) ─      │
│  ① 배움   다이제스트 4채널(시그널·트랙·프롬프트·어학)    │
│  ② 실행   원클릭 "실행의 궤도에 추가" → 오늘 할 일       │
│  ③ 복기   완료 체크 + 한 줄 회고                       │
└──────────────────────────────────────────────────────┘
```

- **배움 4채널 전부**가 궤도와 연결됩니다. 테크 트랙의 액션 챌린지는 기존 `add-orbit`으로, 나머지 3채널(시그널·AI 프롬프트·어학)은 기존 커스텀 궤도 API 재사용으로 — 역시 신규 API 0입니다.
- **루프 리포트는 두 뷰에 공통 상주**하는 다리입니다. 완료율·연속·집중 도메인 타일을 누르면 기록 뷰의 해당 주차로 드릴다운됩니다.
- 인증 상태는 비로그인 / 콜드 스타트 / 궤도 보유 / 이번 주 미개시의 **4상태 매트릭스**로 정밀 분기해, 각 상태마다 다음 행동이 명확한 화면을 보여줍니다.

재미있는 발견도 있었습니다. Pace Note의 기본 궤도 5종("AI 스터디 프롬프트 직접 실행해보기", "비즈니스 일본어 문장 3번 읽기"…)은 사실상 이 채널-궤도 매핑의 **하드코딩판**이었습니다. 대통합은 이를 "오늘의 실제 콘텐츠에서 동적으로 추가"로 승격시킨 것이고, 기본 궤도는 콜드 스타트의 폴백으로 남았습니다.

## 6. 🌅 승계 실행: 3단계 일몰

패리티 게이트가 해소된 뒤, 승계는 3단계로 집행했습니다.

| Phase | 내용 |
| :--- | :--- |
| **Phase 1** | 리런 자체 아카이브 상세(`/relearn/daily/:date`) 구축 — 과거 날짜 열람의 마지막 의존을 제거하고, 내부 링크를 전부 신규 경로로 전환 |
| **Phase 2** | 진입점 교체 — GNB·홈·이메일의 진입을 리런으로 전환하고, 구 경로(`/daily`, `/pacenote`)는 **서버 301**로 SEO를 이관 |
| **Phase 3** | 구 페이지 코드·라우트·메타·문서 위상 정리 — 두 개의 대형 페이지 컴포넌트가 리포지토리에서 퇴역 |

주변 정합도 함께 마쳤습니다. 홈의 서비스 섹션은 Daily·Pace 두 카드를 리런 플래그십 카드 하나로 병합했고, Admin 콘솔은 리런 체제에 맞춰 그룹 4→3·탭 10→8로 재편했으며, 매일 발송되는 뉴스레터는 "📬 ReLearn Daily"로 브랜딩을 승계했습니다.

## 7. 🎯 마치며: 통합은 뺄셈이었다

이번 대통합에서 배운 것을 한 문장으로 줄이면 이렇습니다 — **좋은 통합은 덧셈이 아니라 뺄셈이다.**

- 신규 API·신규 잡·신규 컬러 토큰 전부 **0**
- 모놀리식 2페이지 → 재사용 섹션 컴포넌트로 해체(부채 상환)
- 메뉴 2개 → 여정 1개, 그러나 URL과 SEO 자산은 100% 보존
