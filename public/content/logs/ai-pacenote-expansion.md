# 고인물은 어떻게 AI를 망치는가? 150개 추천 풀 한계 돌파기

인공지능을 도입할 때 수많은 메이커들이 저지르는 가장 흔한 착각은 **"AI는 일단 연동해 두기만 하면 알아서 훌륭하게 굴러갈 것이다"**라는 낙관주의입니다.

초기 PriSincera의 Pace Note 맞춤형 목표 가이드는 단순하고 직관적이었습니다. 유저가 대시보드를 열면 백엔드(`pacenote-api.mjs`) 내부에 정적으로 하드코딩해 둔 15개의 궤도 풀에서 무작위로 꺼내어 제시하는 방식이었죠. 

그러나 이 방식은 오래가지 못했습니다. 매일 대시보드를 찾는 하드 유저들이 늘어나며 **"맨날 보던 목표만 계속 나온다"**는 고인물 현상이 발생했고, 정형화된 데이터의 반복은 사용자들의 의욕을 급격히 갉아먹었습니다. 

우리는 이 정체된 AI 엔진에 무한한 생동감을 공급하고, 유저의 실제 반응 데이터를 수렴해 스스로 진화하는 **"데이터 기반 피드백 플라이휠(Data-Driven Feedback Flywheel)"**을 전격 도입하여 이 한계를 돌파하기로 결정했습니다.

## 🚨 조우한 문제: 고착화된 추천이 가져오는 종말

하드코딩을 타파하기 위해 우리는 매일 자정 인공지능이 새로운 추천 목표를 동적으로 창조하여 공급해 주는 파이프라인(`pacenote-composer.mjs`)을 구축하고 데이터베이스(`config/pacenote_daily_pool`)로 분리했습니다.

더 이상 배포 없이도 매일 신선한 아이템들이 쏟아져 나오게 되었지만, 여기에는 또 다른 숨겨진 기술적 복병이 있었습니다. 
유저들에게 풍부한 선택지를 제공하되 데이터베이스의 탐색 딜레이와 비용 효율성을 확보하기 위해서는 추천 풀의 크기를 **최대 150개**로 제한(Cap)해야 했습니다. 

매일 새 궤도 3개가 추가되므로, 자연스럽게 기존 궤도 3개를 밀어내고 삭제해야 하는 **"퇴출(Eviction)"** 시점이 찾아온 것입니다.

단순하게 '오래된 순(FIFO)'으로 삭제하면 어떻게 될까요? 유저들의 반응이 폭발적이었던 훌륭한 명작 궤도들이 생성일이 지나 영구 삭제되는 큰 손실이 발생합니다.
반대로 '인기 없는 순(Picks가 적은 순)'으로만 삭제하면 어떻게 될까요? 오늘 막 태어난 신생 궤도들은 노출 시간이 짧아 누적 선택률이 낮을 수밖에 없고, 결국 태어나자마자 즉사해 버리는 불공정한 학살이 벌어집니다. 

결국 기득권을 쥔 초기 명작들만 영원히 살아남아 풀을 독점하고 신생 아이디어는 전멸하여, 추천 목록이 완벽하게 고여버리는 **"AI 추천 고착화(Stagnation)"**라는 기술적 한계에 부딪혔습니다.

## 🏗️ 고인물 방지형 동적 퇴출 알고리즘 (Dynamic Eviction Algorithm)

우리는 이 모순을 극복하기 위해 물리 법칙에서 영감을 얻은 **"속도(Velocity) 기반 다차원 퇴출 알고리즘"**을 설계했습니다.

### 1. Velocity (일평균 선택 속도) 공식
시간이라는 변수에 의해 왜곡되는 누적 수치를 보정하기 위해, 아이템이 생성되어 풀에 존재한 생존일 수 대비 실제 유저가 선택한 횟수를 정규화한 **Velocity** 점수를 매일 밤 계산합니다.

$$Velocity = \frac{Total\ Picks}{Active\ Days}$$

*   $Total\ Picks$: 해당 궤도가 유저들에 의해 선택된 누적 횟수
*   $Active\ Days$: 궤도가 데이터베이스에 처음 들어온 시점부터 현재까지 경과한 일수

### 2. 신생 궤도 보호령 (Grace Period) 및 수명 제한 (TTL)
이 Velocity 공식을 기반으로 하되, 알고리즘의 맹점을 커버하기 위해 두 가지 추가적인 안전장치를 장착했습니다.

*   **생성 유예 기간(Grace Period - 7일)**:
    새로 생성된 지 $Active\ Days \le 7$ 미만인 신생 궤도는 유저에게 노출될 충분한 기회를 보장받기 위해 Velocity 점수와 무관하게 **퇴출 대상에서 영구 제외(보호)**합니다.
*   **강제 퇴출 수명(TTL - 45일)**:
    아무리 인기가 폭발해 Velocity 점수가 압도적인 최상위 궤도라 할지라도, $Active\ Days \ge 45$가 지나면 "훌륭하게 임무를 마친 명작"으로 판단하여 **강제로 강단에서 은퇴(강제 퇴출)**시킵니다. 이는 생태계의 영구적인 신선함을 유지하는 절대 방어선입니다.

매일 자정, 150개의 Cap을 초과하는 순간 알고리즘이 유예 기간(7일 이하)이 지난 대상들 중 **Velocity 점수가 가장 낮은 하위 3개 궤도를 정밀 요격하여 퇴출**함으로써, 추천 생태계의 생동감 넘치는 자정 작용을 이끌어 냈습니다.

## 🔄 스스로 진화하는 AI 피드백 플라이휠 (Feedback Flywheel)

이 구조의 진정한 백미는 단순히 퇴출을 자동화한 것에 그치지 않고, 잘 굴러간 유저 반응 데이터가 다시 AI 학습의 밑거름으로 환류되는 **순환형 구조**에 있습니다.

```text
                  [ User Interface: Pace Note Dashboard ]
                                     |
                          (Choose / Complete Data)
                                     v
                       [ Users' Actions Database ]
                                     |
                            (Extract Top Picks)
                                     v
  +--------------------> [ 150 Daily Pool ] <--------------------+
  |                                                             |
  | (Dynamic Few-Shot)                                (Evict Velocity < 0.1)
  |                                                             |
  v                                                             v
[ Gemini API Composer ] <-------------------------------- [ Auto-Eviction Job ]
```

### Dynamic Few-Shot Prompting 기법
매일 자정 크론 잡(`pacenote-composer.mjs`)이 구동될 때, 알고리즘은 수집된 전체 데이터베이스에서 **"최소 2번 이상 선택받은 궤도 중 Velocity 점수가 최상위인 3가지 궤도"**를 정밀 색출합니다. 이를 우리는 **"명예의 전당(Hall of Fame)"**으로 부릅니다.

인공지능 모델(Gemini Flash)이 내일 배포할 신규 궤도를 작명하고 설계하기 직전, 우리는 시스템 프롬프트에 이 명예의 전당에 오른 3가지 실제 궤도 데이터를 **"현재 실제 유저들에게 폭발적인 동기부여를 일으킨 최고의 트렌디 추천 사례"**로 주입합니다.

이 동적 Few-Shot 기법을 탑재한 결과:
*   AI가 별도의 복잡하고 비용이 비싼 미세 조정(Fine-Tuning) 훈련을 거치지 않고도, 유저들이 현재 어떤 어조(Tone), 난이도, 형식의 목표 가이드에 반응하고 열광하는지 스스로 **실시간 트렌드를 파악하고 학습**합니다.
*   공급되는 새로운 목표의 품질과 매력도가 시간이 갈수록 우상향하며 정밀해지는 혁신적인 유기적 환류 고리가 완성되었습니다.

## 💡 결론 및 향후 계획

정적인 데이터를 코딩해 놓고 방치해 두는 시스템은 결국 도태됩니다. 
Pace Note에 적용된 **"150개 추천 풀 제한 및 동적 퇴출 알고리즘"**, 그리고 데이터를 AI 프롬프트에 환류하는 **"피드백 플라이휠"**은 끊임없이 움직이며 스스로의 품질을 개선하는 진정한 현대식 데이터 엔지니어링의 정수를 보여줍니다.

우리는 이에 머무르지 않고, 특정 유저의 완료 카테고리 선호도를 분석하여 카테고리 가중치에 비례해 최적의 맞춤 목표를 제공하는 **초개인화 하이브리드 추천 큐레이션** 기술을 개발해 나갈 것입니다. 

메이킹은 단 한 번의 배포로 끝나지 않습니다. 데이터를 수집하고, 정제하고, 그것을 인공지능에 돌려주어 스스로 굴러가게 만드는 것. 이것이 PriSincera가 추구하는 한계 돌파의 엔지니어링입니다.
