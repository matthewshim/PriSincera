# サービス規模拡大に伴うバックエンドの脅威：PriSincera ウェブサービスセキュリティ精密診断と5つの脆弱性撃退記

PriSinceraは、従来の静的SPAとシンプルなGCS（Google Cloud Storage）JSON読み取り専用構造から脱却し、2026年5月に**ExpressバックエンドマルチルーターアーキテクチャおよびFirestoreベースのステートフルシステム**へと完全に高度化されました。

サービス機能の充実とビジネスモデルの拡張は、ビルダーとして非常に刺激的な経験です。しかし、アーキテクチャが洗練され、連携するAPIエンドポイントが増えるにつれて、**攻撃対象領域（Attack Surface）**もそれに比例して拡大します。

最近、PriSinceraのコードベース全体に対して精密なセキュリティ監査を実施し、その過程で発見された**重大な脆弱性から些細だが致命的だったランタイム欠陥まで、合計5つの主要なセキュリティリスクを撃退したエンジニアリングの道のり**を鮮やかに共有します。

## 🏗️ アーキテクチャの進化と新たなセキュリティ責任

セキュリティ脆弱性対策の詳細を見る前に、現在のPriSinceraのマルチルーティングデータアーキテクチャを理解する必要があります。

```text
                        [ React SPA Frontend ]
                                   |
                                   | (Firebase ID Token)
                                   v
                          [ Express Server ]
           +-----------------------v-----------------------+
           |                       |                       |
           v                       v                       v
     [ Admin API Router ]    [ Study API Router ]    [ PaceNote  Router ]
     (Firebase Admin Auth)   (Authentication)        (Authentication)
           |                       |                       |
           +-----------------------v-----------------------+
           |                                               |
           v                                               v
     [ Firestore Database ] <--------------------- [ GCS Storage Fallback ]
     (subscribers, pacenotes,                      (daily_signals, index.json)
     study_progress, email_logs)
```

購読データ、ユーザーごとの学習進捗（芝生）、Pace NoteアクションプランはFirestoreデータベースに統合保存され、バックエンドは`/admin/api`、`/api/study`、`/api/pacenote`ルーターによって物理的に責務を分離しました。

各APIルートで異なる認証レベルとレート制限が必要になったため、既存のフラットなセキュリティ構造を立体的な多層防御システムにアップグレードするという新たな技術的課題に直面しました。

## 🛠️ 私たちが撃退した5つの主要な脆弱性とエンジニアリングパッチ

静的監査および侵入シミュレーションを通じて発見された5つの欠陥と、サービス無停止の状態でこれらを完全に修復した具体的なトラブルシューティング履歴です。

### 1. 一時的なデバッグAPIが残した痕跡：個人情報（PII）漏洩の遮断 (🔴 Critical)
*   **発見された問題**: 過去にローカル開発環境でメール送信状態や購読者状態を便利に確認するために作成されていた一時的なデバッグAPI（`/api/env-check`、`/api/temp-check-subs`、`/api/temp-logs`）が、プロダクション環境にそのまま露出していました。
*   **セキュリティ上の脅威**: 誰でもこのAPIを呼び出すことで、全購読者のメールアドレスリスト（PII、個人情報）を収集したり、メール送信用のSMTPサーバーアカウント情報など、機密性の高いインフラ設定値を奪取したりする可能性があり、GDPRおよび個人情報保護法上、極めて危険な状態でした。
*   **エンジニアリング措置**:
  - 該当するデバッグエンドポイントは**プロダクションコード（`server.mjs`）から完全に削除**しました。
  - 購読者情報の照会およびログ確認が必要な正式な管理機能は、Firebase JWT Admin IDトークンの検証と`super_admin`役割照合ミドルウェアを完全に経由してのみアクセス可能な**管理者専用の安全なルーター（`/admin/api/subscribers`）に完全に移行**しました。

### 2. ブログスラッグ（Slug）パラメータの有効性検証とNoSQLインジェクションの遮断 (🟡 Medium)
*   **発見された問題**: 技術ブログであるBuilder's Logの記事ごとの閲覧数を増加させる`/api/builderslog/:slug/view` APIにおいて、`:slug`パラメータに対する形式制限が一切存在しませんでした。
*   **セキュリティ上の脅威**: 攻撃者がSlug値に特殊記号や上位パスを示す文字（`../`）などを任意に注入することで、データベース内の他の領域のドキュメントを破損させたり、予期せぬシステム誤動作を引き起こしたりする潜在的なインジェクションリスクが存在しました。
*   **エンジニアリング措置**:
  - スラッグパラメータが流入する入口に、英大文字・小文字、数字、ハイフン（`-`）、アンダースコア（`_`）のみを許可する強力な**正規表現検証ミドルウェア**を配置しました。
  - これにより、異常な文字が1文字でも混入した場合、APIが即座にリクエストを拒否する防御壁を構築しました。

```javascript
// 実際に適用された正規表現ベースのSlug安全フィルタリング
const slug = req.params.slug;
if (!/^[a-zA-Z0-9-_]+$/.test(slug)) {
  return res.status(400).json({ error: "Invalid slug format" });
}
```

### 3. ユーザー入力ペイロード長制限によるデータベース過負荷防御 (🟡 Medium)
*   **発見された問題**: Pace Note機能において、ユーザーが自分だけのカスタムタスクを追加する`POST /api/pacenote/add`エンドポイントで、入力される`title`値の文字数に制限がありませんでした。
*   **セキュリティ上の脅威**: 悪意のあるユーザーがメガバイト単位の巨大なテキストや異常なダミーペイロードを乗せてリクエストを乱発した場合、Firestoreのストレージ容量が急激に枯渇したり、サーバーメモリ占有率が急上昇したりして、サービス全体が麻痺（DoS）する危険がありました。
*   **エンジニアリング措置**:
  - カスタムタスク追加APIの段階で、**最大文字数を100文字に強制制限**する防御ロジックを追加しました。
  - これにより、ビジネス要件（タスクタイトル設定）を完全に満たしつつ、異常なペイロード注入攻撃を根本的にフィルタリングする境界線を構築しました。

### 4. Firestore Security Rulesのマッチングエラーの正常化 (🟡 Medium)
*   **発見された問題**: Firestoreのリアルタイムデータクエリを安全に制御する`firestore.rules`設定において、学習進捗コレクション（`study_progress`）に対する権限マッチングパスが誤って設計されていました。
  - *既存のルール*: `match /study_progress/{userId}/{document=**}`
*   **セキュリティ上の脅威**: 既存のルールは`{userId}`のサブコレクションツリーのみを保護するものであり、ユーザーの進捗状況情報が格納されたルートドキュメント自体（`/study_progress/{userId}`）に対するクライアントからの直接の読み書き権限を適切に保護できていなかったため、連携エラーやルール回避の余地が存在しました。
*   **エンジニアリング措置**:
  - 以下のようにセキュリティルールを、ユーザー自身の進捗情報ドキュメント自体とその下位ツリー全体に1対1でマッチするように精巧に再記述し、セキュリティの完全性とクライアントAPI呼び出しの安定性を同時に確保しました。

```javascript
// 変更完了された安全なFirestoreルール
match /study_progress/{userId} {
  allow read, write: if request.auth != null && request.auth.uid == userId;
  match /{document=**} {
    allow read, write: if request.auth != null && request.auth.uid == userId;
  }
}
```

### 5. PriStudy Progress APIの500内部エラー追跡：ランタイムReferenceErrorの解決 (🟡 Medium)
*   **発見された問題**: 学習進捗を読み込む`GET /api/study/progress` API内部で、Eメールデータが失われた旧バージョンの加入者向けにEメールフィールドを自動補正する例外処理ロジックが動作する際に、未宣言の変数`docRef`を参照してサーバークラッシュおよび500エラーを送信していた隠れたバグを発見しました。
*   **エンジニアリング措置**:
  - 関数スコープ内に`const docRef = db.collection(COLLECTIONS.STUDY_PROGRESS).doc(uid);`変数を明確に宣言およびインスタンス化し、バインディングを完了しました。
  - 非同期修正トランザクションがあらゆる状況で例外処理なしにスムーズに動作するよう、欠陥を完全に除去しました。

## 🛡️ バックエンド全体のセキュリティ要塞化：レート制限 & CSP & CORS

特定のバグ解決に留まらず、バックエンドExpressウェブサーバーインフラレベルの防御体系も一層堅固に要塞化しました。

### 1. 細分化されたAPIリクエスト制限（Rate Limiting）
ブルートフォース攻撃や分散サービス拒否（DDoS）攻撃からAPIサーバーを保護するため、`express-rate-limit`モジュールを導入し、ルーターの特性に合わせて速度制限レイアウトを精密に配置しました。
*   **グローバル公開API (`/api/`)**: 1分あたり最大60回 (`apiLimiter`)
*   **購読および購読解除API (`/api/subscribe`、`/api/unsubscribe`)**: 15分あたり最大5回 (`subscribeLimiter` - 不正利用防止)
*   **管理者API (`/admin/api/`)**: 15分あたり最大100回 (`adminLimiter` - 不要なスキャン遮断)

### 2. CORSワイルドカードポリシーの廃止
既存の開発便宜のために緩やかに設定されていたCORSワイルドカード（`*`）設定を全面的に廃止し、私たちが指定したプロダクション公式ドメイン（`https://www.prisincera.com`）から発信された信頼できるHTTPリクエストのみがバックエンド応答を受け入れるように安全に制限しました。

### 3. Helmetベースの厳格なCSP（コンテンツセキュリティポリシー）適用
セキュリティヘッダーを制御する`helmet`ミドルウェアを精密にチューニングし、クロスサイトスクリプティング（XSS）およびクリックジャッキングを根本的に遮断する厳格なモダンCSPルールを確立しました。

```javascript
app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'", "https://apis.google.com", "https://www.gstatic.com"],
      styleSrc: ["'self'", "'unsafe-inline'", "https://fonts.googleapis.com"],
      fontSrc: ["'self'", "https://fonts.gstatic.com"],
      imgSrc: ["'self'", "data:", "https:"],
      connectSrc: ["'self'", "https://identitytoolkit.googleapis.com", "https://securetoken.googleapis.com"],
      frameSrc: ["'self'", "https://*.firebaseapp.com", "https://prisincera.firebaseapp.com"],
      objectSrc: ["'none'"],
      baseUri: ["'self'"]
    }
  },
  strictTransportSecurity: { maxAge: 31536000, includeSubDomains: true, preload: true }
}));
```

## 🐳 インフラおよびコンテナデプロイレベルのセキュリティ

コンテナがハッキングされてもホストサーバー全体の権限が乗っ取られないよう、**運用段階のセキュリティ高度化**も怠りませんでした。

### 1. Docker非ルート（Non-Root）アカウント実行構造への転換
従来の軽量なNodeコンテナはデフォルトのルート（Root）権限でプロセスを実行しており、万が一コンテナ脱出の脆弱性が発見された場合、ホストLinuxサーバー全体が乗っ取られる危険性がありました。
これを防ぐため、`Dockerfile`内部に`prisincera`専用のシステムアカウント（UID 1001）とグループを作成し、サーバー起動プロセスが該当の非ルート権限下でのみ制限的に実行されるようビルドパイプライン設定を修正しました。

### 2. GCP Secret Managerを介した完全非物理的資格情報管理
ソースコードにAPIキーがハードコーディングされないようにすることは基本です。私たちはこれに加えて、機密性の高い外部API認証トークン（`GEMINI_API_KEY`、`GITHUB_TOKEN`）をサーバー起動環境変数テキストファイルにも直接記載せず、デプロイパイプラインで**Google Cloud Secret Manager**を通じてランタイムに動的に注入されるように設計することで、認証情報漏洩経路を完全に遮断しました。
