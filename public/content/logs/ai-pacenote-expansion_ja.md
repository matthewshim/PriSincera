# ベテランユーザーはAIをいかに停滞させるか？ 150個の推薦プール限界突破記

AIを導入する際、多くのメーカーが陥る最も一般的な錯覚は、**「AIはひとたび連携させれば、勝手にうまく機能するだろう」**という楽観主義です。

初期のPriSinceraのPace Noteパーソナライズ目標ガイドは、シンプルで直感的でした。ユーザーがダッシュボードを開くと、バックエンド(`pacenote-api.mjs`)内部に静的にハードコーディングされた15個のトラックプールから無作為に取り出して提示する方式でした。

しかし、この方式は長くは続きませんでした。毎日ダッシュボードを訪れるハードユーザーが増えるにつれ、**「いつも同じ目標ばかり出てくる」**というベテランユーザー現象が発生し、定型化されたデータの繰り返しはユーザーの意欲を急速に削ぎました。

私たちは、この停滞したAIエンジンに無限の生命力を供給し、ユーザーの実際の反応データを収斂して自ら進化する**「データ駆動型フィードバックフライホイール(Data-Driven Feedback Flywheel)」**を本格的に導入することで、この限界を突破することに決めました。

---

## 🚨 遭遇した問題：固定化された推薦がもたらす終焉

ハードコーディングを打破するため、私たちは毎日深夜にAIが新しい推薦目標を動的に生成・供給するパイプライン(`pacenote-composer.mjs`)を構築し、データベース(`config/pacenote_daily_pool`)に分離しました。

もはやデプロイなしでも毎日新鮮なアイテムが溢れ出るようになりましたが、ここにはまた別の隠れた技術的伏兵がありました。
ユーザーに豊富な選択肢を提供しつつ、データベースの探索遅延と費用効率を確保するためには、推薦プールのサイズを**最大150個**に制限(Cap)する必要がありました。

毎日新しいトラックが3つ追加されるため、自然と既存のトラック3つを押し出して削除しなければならない**「退去(Eviction)」**の時が訪れたのです。

単に「古い順(FIFO)」で削除するとどうなるでしょうか？ ユーザーの反応が爆発的だった素晴らしい名作トラックが、生成日が過ぎたために永久削除されるという大きな損失が発生します。
逆に「人気のない順(Picksが少ない順)」だけで削除するとどうなるでしょうか？ 今日生まれたばかりの新生トラックは、露出時間が短いため累積選択率が低くなるしかなく、結局生まれてすぐに即死してしまう不公平な虐殺が起こります。

結局、既得権を持つ初期の名作だけが永遠に生き残りプールを独占し、新生アイデアは全滅し、推薦リストが完全に停滞してしまう**「AI推薦固定化(Stagnation)」**という技術的限界にぶつかりました。

## 🏗️ 停滞防止型動的退去アルゴリズム (Dynamic Eviction Algorithm)

私たちはこの矛盾を克服するため、物理法則からインスピレーションを得た**「速度(Velocity)ベースの多次元退去アルゴリズム」**を設計しました。

### 1. Velocity (一日平均選択速度) 公式
時間という変数によって歪められる累積数値を補正するため、アイテムが生成されてプールに存在した生存日数に対する、実際のユーザーが選択した回数を正規化した**Velocity**スコアを毎晩計算します。

$$Velocity = \frac{Total\ Picks}{Active\ Days}$$

*   $Total\ Picks$: 当該トラックがユーザーによって選択された累積回数
*   $Active\ Days$: トラックがデータベースに初めて入ってから現在までの経過日数

### 2. 新生トラック保護令 (Grace Period) および寿命制限 (TTL)
このVelocity公式を基盤としつつ、アルゴリズムの盲点をカバーするために2つの追加的な安全装置を装備しました。

*   **生成猶予期間(Grace Period - 7日)**:
    新しく生成されてから $Active\ Days \le 7$ 未満の新生トラックは、ユーザーに露出される十分な機会を保障するため、Velocityスコアに関わらず**退去対象から永久に除外(保護)**します。
*   **強制退去寿命(TTL - 45日)**:
    どんなに人気が爆発しVelocityスコアが圧倒的に最上位のトラックであっても、$Active\ Days \ge 45$ が過ぎれば、「見事に任務を終えた名作」と判断し、**強制的にステージから引退(強制退去)**させます。これは生態系の永続的な新鮮さを維持する絶対防衛線です。

毎日深夜、150個のCapを超過する瞬間、アルゴリズムは猶予期間(7日以下)が過ぎた対象の中から**Velocityスコアが最も低い下位3つのトラックを精密に選別して退去**させることで、推薦生態系の生動感あふれる自浄作用を導き出しました。

## 🔄 自ら進化するAIフィードバックフライホイール (Feedback Flywheel)

この構造の真髄は、単に退去を自動化したことに留まらず、うまく運用されたユーザー反応データが再びAI学習の基礎として還流される**循環型構造**にあることです。

```text
                  [ User Interface: Pace Note Dashboard ]
                                     |
                          (Choose / Complete Data)
                                     v
                       [ Users' Actions Database ]
                                     |
                            (Extract Top Picks)
                                     v
  +--------------------> [ 150 Daily Pool ] <--------------------+
  |                                                             |
  | (Dynamic Few-Shot)                                (Evict Velocity < 0.1)
  |                                                             |
  v                                                             v
[ Gemini API Composer ] <-------------------------------- [ Auto-Eviction Job ]
```

### Dynamic Few-Shot Prompting 技術
毎日深夜、クロンジョブ(`pacenote-composer.mjs`)が駆動される際、アルゴリズムは収集された全データベースから**「最低2回以上選択され、Velocityスコアが最上位の3つのトラック」**を精密に抽出します。これを私たちは**「殿堂入り(Hall of Fame)」**と呼びます。

AIモデル(Gemini Flash)が翌日デプロイする新規トラックを命名・設計する直前、私たちはシステムプロンプトに、この殿堂入りした3つの実際のトラックデータを**「現在、実際のユーザーに爆発的なモチベーションを引き起こしている最高のトレンディな推薦事例」**として注入します。

この動的Few-Shot技術を搭載した結果：
*   AIが別途の複雑で費用のかかる微調整(Fine-Tuning)訓練を経ることなく、ユーザーが現在どのようなトーン、難易度、形式の目標ガイドに反応し熱狂しているかを自ら**リアルタイムでトレンドを把握し学習**します。
*   供給される新しい目標の品質と魅力度が時間とともに右肩上がりに向上し、精密化される革新的な有機的還流サイクルが完成しました。

## 💡 結論および今後の計画

静的なデータをコーディングして放置するシステムは、結局は淘汰されます。
Pace Noteに適用された**「150個の推薦プール制限および動的退去アルゴリズム」**、そしてデータをAIプロンプトに還流する**「フィードバックフライホイール」**は、絶えず動き、自らの品質を改善する真の現代的データエンジニアリングの精髄を示しています。

私たちはこれに留まらず、特定のユーザーの完了カテゴリ選好度を分析し、カテゴリの重みに比例して最適なカスタム目標を提供する**超パーソナライズドハイブリッド推薦キュレーション**技術を開発していく予定です。

メイキングは一度のデプロイで終わりません。データを収集し、精製し、それを人工知能に返し、自ら動くようにすること。これがPriSinceraが追求する限界突破のエンジニアリングです。
