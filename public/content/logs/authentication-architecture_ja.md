# Googleワンクリックログインの隠された落とし穴：ユーザーと管理者セッションを完全に隔離する

サービスを運営する上で、サインアップのコンバージョン率（Conversion Rate）を劇的に向上させる最も強力なカードの一つは、間違いなく**「Googleワンクリックログイン（Google OAuth）」**です。

PriSinceraもまた、ユーザーが目標を設定し実践する上で生じるあらゆる煩わしい障壁（Friction）をゼロにするため、従来のPriStudy時代のメール/パスワードベースの会員登録システムを大胆に廃止し、Google単一認証への全面的な世代交代を実現しました。

ユーザーはワンクリックでサービスにすぐに定着し、私たちの成長フライホイールは完璧に機能しているように見えました。しかし、この洗練された入り口のすぐ後ろには、サービスの成長を脅かす**「セキュリティセッションの衝突」**という致命的な落とし穴が隠されていました。

---

## 🚨 遭遇した問題：ローカルストレージの覇権争い

認証システムを単一化する過程で、私たちのアーキテクチャ上のアキレス腱として浮上したのは、**管理者バックオフィス（`AdminDashboard`）**でした。

セキュリティ上の理由から、このバックオフィスはユーザーデータを全体的に照会・管理する強力な権限を持つため、一般的なGoogleログインではなく、**事前に承認されたセキュリティ管理者アカウント（メール/パスワード）**でのみログインが可能である必要がありました。

しかし、ブラウザ環境で動作するFirebase Authモジュールは、基本的に**単一インスタンス**でセッションを維持します。つまり、ブラウザのローカルストレージには、ただ一つのセッション情報しか保存できないという物理的な制約が存在しました。

これにより、奇妙なバグが発生しました：
1. 一般ユーザーが自身のGoogleアカウントでログインを完了します。
2. この状態で管理者（アドミン）ダッシュボードにアクセスし、管理者アカウントでメールログインを試みます。
3. ローカルストレージ上のGoogleセッショントークンが、管理者セッショントークンに強制的に上書きされます。
4. ユーザーのブラウザタブに戻るとセッションがねじれてしまい、アカウントが管理者権限として誤認識されたり、権限なしとして強制ログアウト（弾き出される）現象が頻発するようになりました。

これは、一つのブラウザ内で「一般ユーザー」と「セキュリティ管理者」という完全に異なる二つの役割を隔離して共存させることができない、深刻なセッションの絡み合い事態でした。

## 🏗️ セッション隔離アーキテクチャ (Session Isolation Architecture)

私たちはブラウザレベルでセッショントークンを完全に隔離するため、Firebaseの多重アプリケーション初期化技術を導入し、**セッションを物理的に分離する二重セキュリティトラック**を設計しました。

```text
  +-------------------------------------------------------------------------+
  |                      [ Client Browser LocalStorage ]                    |
  |                                                                         |
  |  Default App Session (Google Login) ----> ID Token ----> User API       |
  |  Admin App Session (Email / PW) --------> Admin Token -> Admin API      |
  +-------------------------------------------------------------------------+
                                       |
                                       v
  +-------------------------------------------------------------------------+
  |                      [ Backend Security Gateway ]                       |
  |                                                                         |
  |  User API (/api/study) ---------> firebase-admin.verifyIdToken()        |
  |  Admin API (/admin/api) --------> Super Admin Role Verification         |
  +-------------------------------------------------------------------------+
```

### 1. 複数のFirebase Appの初期化（`ADMIN_APP`の分離）
一般ユーザーがアクセスするすべての画面では、既存の`[DEFAULT]` Firebaseインスタンスを通じてGoogleソーシャル認証セッションを維持するように限定しました。
そして、セキュリティ領域である`AdminDashboard.jsx`内部に進入する際は、別途の名前を付けて管理者専用の独立したインスタンスをその場で多重起動させました。

```javascript
// AdminDashboard.jsx内でのセッション隔離初期化
import { initializeApp, getApp, getApps } from 'firebase/app';
import { getAuth } from 'firebase/auth';

const firebaseConfig = { /* ... */ };

// DEFAULTインスタンスと衝突しないよう、明示的に隔離されたADMIN_APPネームスペースを確保
const adminApp = getApps().find(app => app.name === 'ADMIN_APP') 
  || initializeApp(firebaseConfig, 'ADMIN_APP');

const adminAuth = getAuth(adminApp);
```

この高度な隔離技術により、Firebaseはブラウザのローカルストレージに`firebase:authUser:...:[DEFAULT]`キーと`firebase:authUser:...:ADMIN_APP`キーを**完全に独立した固有のレコードとして物理的に分離・保存**するようになりました。一般ユーザーとしてGoogleログインを維持している状態でも、バックオフィス管理者セッションが上書きされることなく、平和かつ堅牢に共存できるようになった瞬間でした。

### 2. バックエンドセキュリティミドルウェアの要塞化
認証セッションがクライアント側で精巧に隔離されたため、バックエンドAPIゲートウェイもこれに合致する立体的な多層防御網を稼働させました。

*   **ユーザー専用ルーター（`pacenote-api.mjs`、`study-api.mjs`）：**
    フロントエンドAPI呼び出し時、ヘッダーに流入したGoogle認証ベースのFirebase ID Tokenの整合性を`verifyIdToken()`でリアルタイムに検証します。
    ```javascript
const decodedToken = await admin.auth().verifyIdToken(idToken);
req.user = { uid: decodedToken.uid };
    ```
    この検証を通じて取得した固有の`uid`以外のユーザーデータにはクエリが流入しないよう、データ隔離を保証します。

*   **管理者専用ルーター（`admin-api.mjs`）：**
    `ADMIN_APP`インスタンスで検証された特殊な管理者トークンのみを許可し、流入したセッションがデータベースの`super_admin`ロールグループ照合フィルタリングミドルウェアを無事に通過して初めて、ダッシュボード内部の機密性の高い書き込み（Write）権限を承認します。

## 💡 教訓：ビルダーの視点から振り返るセッション設計

単に「ライブラリがサポートするデフォルトのログイン関数を使用すること」は非常に簡単です。しかし、サービスアーキテクチャが徐々に進化し、**バックオフィス、cronパイプライン、サードパーティ統合**など、様々な権限レベルが絡み合う瞬間、ローカルセッションストレージ一つでさえも深く制御する必要があるセキュリティ責任が伴います。

AuthContextの集中管理を通じてドメイン全体のGoogleログインを円滑に制御し、複数のインスタンスを通じて管理セッションを物理的に隔離することに成功した今回の設計パターンは、将来Appleログインや新しいB2B連携セッションを追加する際にも、コード変更なしで拡張できる**柔軟で非破壊的な高隔離セキュリティ基盤**を提供します。
