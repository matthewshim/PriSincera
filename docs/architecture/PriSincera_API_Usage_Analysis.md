# 📊 PriSincera API Usage Analysis: AI Commit Auto-Matching

본 문서는 Builder's Log 작성 시, 아티클 본문과 가장 밀접하게 연관된 GitHub 커밋 내역을 AI가 자동 추출하도록 설계된 파이프라인의 **API 한도 및 과금 안정성**을 검증한 분석 보고서입니다.

## 1. 개요 (Background)

현재 Builder's Log는 특정 기능이나 파이프라인을 구축한 뒤, 그 과정을 아티클(기술 블로그)로 배포합니다. 이 과정에서 아티클의 내용과 연관된 GitHub 커밋(Commit) 리스트를 추출해 카드 하단에 디자인 요소로 렌더링하고 있습니다.
기존에는 "가장 최근 30개의 커밋"만 무지성으로 불러왔으나, 오랜 기간 개발된 레포지토리(총 385개 커밋) 특성상 과거의 작업 내역에 대한 아티클을 작성할 때는 매칭이 실패하는 병목 현상이 있었습니다. 이를 해결하기 위해 커밋 수집량 한도를 대폭 상향하는 방안을 검토했습니다.

## 2. API 의존성 및 한도 제약 (Free Tier 기준)

현재 Admin 퍼블리싱 파이프라인은 두 가지 외부 API를 사용합니다.

### A. Gemini 2.5 Flash API (Google AI Studio)
- **무료 제공 한도:**
  - **TPM (Tokens Per Minute):** 1,000,000 (1백만)
  - **RPM (Requests Per Minute):** 15
  - **RPD (Requests Per Day):** 1,500
- **토큰 사용량:**
  - 커밋 1개 (`{"type":"feat", "hash":"1a2b3c", "msg":"..."}`) ≈ **약 10 ~ 15 토큰**
  - 마크다운 본문 (평균 2,000자) ≈ **약 2,000 ~ 3,000 토큰**

### B. GitHub REST API (Octokit)
- **한도:** `per_page` 최대 100개 (1회 호출 기준)
- **무료 제공 한도:** Personal Access Token 인증 시 **시간당 5,000회**

## 3. 전사적(Cross-Service) 파이프라인 충돌 시나리오 검증

단순 Admin 배포뿐만 아니라, 현재 구글 클라우드 스케줄러(Cloud Scheduler)에서 구동 중인 데일리 자동화 파이프라인들과의 트래픽 충돌(Rate Limit) 가능성을 검증했습니다.

### 🕒 백그라운드 파이프라인 구동 스케줄
1. **PaceNote (00:00 KST):** 궤도 주입 및 신규 생성 (약 2~3회 호출)
2. **PriStudy (04:00 KST):** 오늘의 1문장 생성 (약 2회 호출)
3. **Daily Digest (08:00 KST):** 뉴스레터 큐레이션 및 스코어링 (약 10~15회 호출)

### 💥 최악의 충돌 시나리오 (Worst-Case)
- **상황:** 아침 8시 정각에 뉴스레터 스코어링이 도는 타이밍에 맞춰, 관리자가 Admin 페이지에서 `385개` 전체 커밋 리스트를 동봉하여 Builder's Log AI 분석을 요청함.
- **예상 토큰 소모량:**
  - Daily Digest 큐레이션 배치 처리: 약 30,000 토큰 소모
  - Admin (본문 + 385개 커밋 전체): 약 10,000 토큰 소모
  - **총합:** 40,000 토큰
- **결론:** 충돌 시에도 **분당 무료 한도(1,000,000 토큰)의 단 4%만 점유**하므로 완벽하게 안전합니다. RPD(일일 한도) 1,500회 역시 하루 총 30회 미만을 사용하는 현재 아키텍처에서는 무한대에 가깝습니다.

## 4. 최종 결정 및 아키텍처 반영 (Action Item)

API 토큰과 호출 한도가 매우 넉넉하게 설계되어 있음이 수치적으로 증명되었습니다. 
속도 지연(API 다중 호출에 따른 Latency)을 최소화하면서도 매칭 정확도를 극대화할 수 있도록, **단일 GitHub API 호출의 최대치인 100개로 한도를 대폭 상향**합니다.

*   **변경 전:** `octokit.rest.repos.listCommits({ per_page: 30 })`
*   **변경 후:** `octokit.rest.repos.listCommits({ per_page: 100 })`

이를 통해, "최근 수개월 치의 개발 내역"이 한 번의 요청에 즉각적으로 프롬프트에 주입되어, 과거에 개발된 기능에 대한 블로그를 쓰더라도 AI가 빠르고 정확하게 커밋을 매칭해 낼 수 있게 되었습니다.
